# CodeAtlas Merge Request Review Framework

CodeAtlas can use generated maps, facts, rules, requirements, and context packs to review merge requests for behaviour changes, contract breaks, rule drift, security/permission issues, test gaps, and architecture drift.

This framework is designed for GitLab workflows using `glab`.

## Safety principle

CodeAtlas must default to **draft-only review**.

It may generate findings, inline comment drafts, review summaries, and `glab` command suggestions, but it must not publish review comments unless the user explicitly approves.

## Required AI-generated header

Every published AI-generated review comment must start with this exact header:

```text
✦ AI GENERATED REVIEW
```

Recommended full header:

```text
✦ AI GENERATED REVIEW
Generated by CodeAtlas from atlas/knowledge, rules, maps, and changed-file evidence. Please verify before relying on this finding.
```

## Review modes

| Mode | Behaviour |
|---|---|
| `draft` | Generate review files only. Do not call `glab` write commands. Default. |
| `prepare` | Generate review files and exact `glab` commands/scripts, but do not execute them. |
| `approved-post` | Post comments only when explicit approval is present. |

Suggested environment variables:

```bash
export CODEATLAS_REVIEW_MODE=draft
export CODEATLAS_POST_REVIEW_COMMENTS=false
export CODEATLAS_REVIEW_APPROVED=false
```

To allow posting after human review:

```bash
export CODEATLAS_REVIEW_MODE=approved-post
export CODEATLAS_POST_REVIEW_COMMENTS=true
export CODEATLAS_REVIEW_APPROVED=true
```

## Inputs

MR review should read:

```text
atlas/knowledge/manifest.yaml
atlas/knowledge/indexes/*.yaml
atlas/knowledge/cards/*.json
atlas/knowledge/nodes/*.yaml
atlas/knowledge/edges.yaml
atlas/knowledge/audit/*.md
atlas/context-packs/*.md
.kiro/steering/*.md
atlas/map/*.yaml
atlas/facts/*.yaml
atlas/domains/*/*.yaml
```

It should also inspect MR data via `glab` where available:

```bash
glab mr view <iid>
glab mr diff <iid>
glab mr changes <iid>
glab mr commits <iid>
```

If a command is unavailable in the installed `glab` version, use the closest equivalent or `glab api` read-only calls.

## Review focus areas

CodeAtlas MR reviews should check:

1. changed files mapped to domains
2. impacted Code Map nodes
3. impacted technical facts
4. impacted technical rules
5. impacted business rules
6. impacted user stories and acceptance criteria
7. frontend/backend API contract drift
8. schema/request/response mismatches
9. permission/auth regressions
10. validation drift
11. error-handling gaps
12. state-transition changes
13. side-effect changes
14. test coverage gaps
15. contradictions introduced or resolved
16. stale generated Atlas artifacts

## Inline comment policy

Prefer inline comments when the finding is tied to a specific changed line or file hunk.

Use a general MR summary comment when:

- the issue spans multiple files
- the finding is architectural
- the finding is about missing tests across a flow
- the finding is about generated Atlas artifacts or documentation
- exact changed-line position is unavailable

Inline comments should be concise and actionable.

## Comment severity

Use these severities:

```text
blocker
major
minor
nit
question
positive
```

Default posture:

- Blockers: auth/security regression, data leak, broken API contract, deleted approved behaviour, invalid generated artifacts.
- Major: likely behaviour regression, missing backend enforcement, high-risk missing tests.
- Minor: maintainability issue, weak naming, incomplete generated mapping.
- Nit: style or cleanup only.
- Question: unclear intent or missing evidence.
- Positive: useful acknowledgement when a change improves rule alignment or removes contradiction.

## Comment shape

Every drafted or posted comment should use this shape:

```markdown
✦ AI GENERATED REVIEW

**Severity:** major  
**Area:** business-rule drift / API contract / permission / validation / tests  
**CodeAtlas evidence:** `br.<domain>.<rule>` → `tr.<domain>.<rule>` → `fact.<domain>.<fact>`  

Finding:
<short finding>

Why it matters:
<impact in plain English>

Suggested action:
<specific fix or verification step>
```

## Output files

MR review should write:

```text
atlas/reviews/mr-<iid>/review-summary.md
atlas/reviews/mr-<iid>/inline-comments-draft.md
atlas/reviews/mr-<iid>/general-comment-draft.md
atlas/reviews/mr-<iid>/glab-readonly-commands.sh
atlas/reviews/mr-<iid>/glab-post-commands.sh
atlas/reviews/mr-<iid>/impacted-atlas-nodes.yaml
atlas/reviews/mr-<iid>/test-gap-findings.yaml
atlas/reviews/mr-<iid>/rule-delta-findings.yaml
atlas/reviews/mr-<iid>/contract-findings.yaml
```

`glab-post-commands.sh` must be generated but must not be run unless approval is explicit.

## Approval gate

Before posting comments, CodeAtlas must ask the user to review the draft pack and confirm posting.

For non-interactive runs, posting is allowed only when all are true:

```text
CODEATLAS_REVIEW_MODE=approved-post
CODEATLAS_POST_REVIEW_COMMENTS=true
CODEATLAS_REVIEW_APPROVED=true
```

If any flag is missing or false, do not post. Generate draft artifacts only.

## `glab` guidance

Use read-only commands freely:

```bash
glab mr view "$MR_IID"
glab mr diff "$MR_IID"
glab mr commits "$MR_IID"
```

For comments, prefer GitLab discussions/inline threads when line positions are available. If the exact inline API position is uncertain, generate the command in `glab-post-commands.sh` with a note requiring human verification instead of executing it.

Never post generic comments for findings that should be inline unless inline positioning is impossible.

## Review summary requirements

`review-summary.md` must include:

- MR title and IID
- review mode
- changed files
- impacted domains
- impacted Atlas nodes
- findings by severity
- proposed inline comments count
- proposed general comments count
- tests that should be added or updated
- whether generated Atlas artifacts need a targeted rerun
- explicit posting status: `not posted`, `posted`, or `posting skipped`

## Non-goals

The MR reviewer should not:

- rewrite the MR author's code unless explicitly asked
- post without approval
- fabricate line positions
- require full CodeAtlas regeneration for every MR
- treat generated Atlas rules as infallible when reverse verification says they are weak
